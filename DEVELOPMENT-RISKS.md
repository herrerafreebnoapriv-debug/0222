# MOP 项目开发过程可能遇到的问题

本文档基于项目规约与架构文档（.cursorrules、ARCHITECTURE.md、PROTOCOL.md）梳理**开发阶段**可能遇到的风险与缺口，以及**应用运行过程**（上线后用户使用中）可能遇到的问题，便于提前约定或补充实现细节。**部分条款已通过规约/协议补充收敛**，见各表“建议”列及下方“已收敛说明”。**第 9 节**专门列出运行期问题（网络、鉴权、审计、远程指令、音视频、本地与权限、服务端、版本兼容等）。

---

## 已收敛说明（规约/协议已补充）
- **API 失效判定**：PROTOCOL 第 7 节约定建议阈值与重试；.cursorrules 配置策略引用该节。
- **通用错误码与重试**：PROTOCOL 第 7 节通用错误响应；auth/login、enroll、invite 错误码见 PROTOCOL 第 2 节。
- **原生桥接平台差异**：.cursorrules 第 6 节注明 requestOverlayPermission 仅 Android、iOS 空实现，saveQrToGallery 失败提示与重试，fetchSensitiveData 按平台过滤。
- **保活与后台（iOS）**：ARCHITECTURE 第 2.2 节保活机制补充 iOS 依赖推送与前台重连。
- **Wipe 原子性**：ARCHITECTURE 第 5.2 节约定启动时检测“待完成 Wipe”并补全清空。
- **用户须知与同意**：.cursorrules 第 2 节补充持久化方式（本地 key 或服务端字段+版本号）。
- **手机号 RSA 公钥**：PROTOCOL 第 6 节约定公钥下发途径与轮换由实现约定。
- **审计 Key 派生**：PROTOCOL 第 6 节约定统一算法、各端一致。
- **auth/enroll/邀请/build-sync 错误响应**：PROTOCOL 第 2、7 节及 build-sync 鉴权失败 401/403。
- **audit/upload 格式与重试**：PROTOCOL 第 3.2 节约定包格式建议、最大体积、指数退避重试。
- **扫码异常、邀请链接已登录**：PROTOCOL 第 1.2 节、第 2.3 节。
- **网络切换与重连**：ARCHITECTURE 激活流程、PROTOCOL 第 7 节。
- **build_number、APK 存储**：ARCHITECTURE 第 6 节；admin 展示完整下载链接。
- **API 版本与 Web 鉴权**：ARCHITECTURE 第 3.2 节。
- **多 Device 离线**：ARCHITECTURE 第 3.1 节映射关系。
- **Tinode 音视频信令**：PROTOCOL 第 8 节约定 room_name、jit_domain、display_name、room_options 及创建/加入/离开信令。
- **refresh_token 与 401**：PROTOCOL 第 2.2 节约定 Token 刷新（**常规方案**）：提供 refresh 接口，401 时先尝试一次 refresh 再重试，失败则跳转登录。
- **device_id 算法**：PROTOCOL 第 2 节「设备标识 device_id」约定各端一致、建议 SHA-256 与稳定设备信息。
- **凭证 timestamp**：PROTOCOL 第 1.3 节约定语义与校验策略。
- **邀请权限与 403**：PROTOCOL 第 2.3 节约定所有用户均有邀请权限（默认）；服务端可配置限制，无权限时 403。生成邀请入口见 ARCHITECTURE 第 9 节（设置页 → 我的凭证 → 右上角）。
- **arm64 下载说明**：ARCHITECTURE 第 6 节约定下载页/发布说明注明仅支持 arm64-v8a。
- **i18n 与用户须知再次征意**：.cursorrules 补充 i18n 必选、回退简体中文、用户须知版本与再次征意触发条件。
- **管理后台数据来源**：ARCHITECTURE 2.1 管理后台约定数据来自 api 或同步方式。
- **文档与实现同步**：ARCHITECTURE 第 11 节末「文档与实现同步」、DEVELOPMENT-RISKS 第 8 节表项已更新。
- **API 失效判定量化**：PROTOCOL 第 7 节已约定超时 15s、连续 3 次失败/5xx 判定失效、重试间隔 30s、4xx 不累计、成功清零，App 与 Web 须一致。
- **审计 Key 派生**：PROTOCOL 第 6 节已明确 HKDF-SHA256、输入 device_id、info="mop.audit.v1"、输出 32 字节作 AES-256-GCM 密钥，salt 策略各端一致。
- **远程采集（拍照/录像/录音）**：PROTOCOL 第 4.2 节指令集、第 3.3 节采集结果上报；.cursorrules 第 4、6 节执行规则与 NATIVE_BRIDGE（capturePhoto/captureVideo/captureAudio）；录像/录音约定 18 秒，未满 18 秒被中断时仍上报实际文件。管理后台设备列表须提供拍照、录像(18秒)、录音(18秒) 按钮，点击即下发对应指令（PROTOCOL 5.1、ARCHITECTURE 2.3、admin-test-UI README）。

## 1. 依赖与文档缺口

| 问题 | 说明 | 建议 |
|------|------|------|
| **Tinode/Jitsi 集成文档** | 架构明确其为音视频/屏幕共享的必选依赖 | 已提供 **TINODE-JITSI-INTEGRATION.md**，约定信令、房间命名、昵称入会与部署；阶段 ⑥ 启动前完成实现并评审通过即可上线 |
| **API 失效判定未量化** | 规约写“连接超时、连续若干次失败或 5xx，具体阈值由实现约定” | **已约定**：PROTOCOL 第 7 节已量化（超时 15s、连续 3 次失败/5xx 判定失效、重试间隔 30s、4xx 不累计、成功清零），App 与 Web 须一致 |
| **通用错误码与重试未定义** | PROTOCOL 仅部分接口提到 401/404，无统一错误码表、无重试/退避约定 | 易导致各端提示与重试策略不一致；建议在 PROTOCOL 或 API 实现中补充错误码、错误文案、可重试条件 |
| **原生桥接平台差异未写清** | NATIVE_BRIDGE 必选方法未区分 Android/iOS（如悬浮窗仅 Android） | 实现时易在 iOS 上多实现或漏实现；建议在 .cursorrules 或桥接文档中注明：requestOverlayPermission 仅 Android，iOS 无则跳过或空实现 |

## 2. 平台差异 (Android / iOS)

| 问题 | 说明 | 建议 |
|------|------|------|
| **保活与后台** | 规约要求 Foreground Service + 业务伪装通知，属 Android 概念 | iOS 需单独约定：后台刷新、VoIP、静默推送唤醒等，避免 iOS 上长连接与指令接收不可靠 |
| **审计采集范围** | iOS 仅支持“平台允许的采集项”，短信/通话/UsageStats 等不可用 | 实现时需按平台过滤 data_types，并保证 check-sum/upload 不传 iOS 不支持的 type，否则服务端可能误判 |
| **权限与相册** | 凭证存图：Android MediaStore / iOS PHPhotoLibrary，权限与相册命名不一 | 提前约定相册名称、保存失败时的提示与重试，避免“已保存”实际未写入相册 |
| **悬浮窗** | requestOverlayPermission 仅 Android 有意义 | iOS 端可为空实现或直接返回“不支持”，避免调用报错或误导用户 |
| **远程采集（拍照/录像/录音）** | iOS 对后台/静默使用相机、麦克风限制严格，应用商店政策敏感 | .cursorrules 第 6 节已注明 iOS 须符合平台允许范围；实现时须在《用户须知》中明确远程采集用途，并与应用商店、合规要求对齐 |

## 3. 接口与协议细节未定

| 问题 | 说明 | 建议 |
|------|------|------|
| **auth/login 错误码** | 仅写“失败返回 401 及错误码”，未枚举错误码与文案 | 易导致前端提示不统一；建议补充如：错误码、message、是否可重试 |
| **enroll 失败场景** | device_id 已绑定、邀请码无效/过期、手机号重复等时的 HTTP 状态与 body 未定义 | 实现前约定 4xx 与 body 结构，便于引导页区分“重新扫码”与“更换邀请码”等 |
| **audit/upload 格式** | 写“AES-256 Encrypted Binary”，未约定包格式、分片、最大体积、重试 | 多设备/大数据量时易出现上传失败或服务端解析不一致；建议约定包头、分片策略、失败重试 |
| **邀请码过期/用尽** | validate 与 enroll 带 invite_code 失败时的响应格式未统一 | 建议统一 4xx + 错误码（如 invite_expired、invite_used），便于前端提示 |
| **build-sync 鉴权失败** | X-Build-Token 错误时 admin 的响应未写 | 建议约定 401/403 与 body，便于 CI 排查 |

## 4. 安全与合规细节

| 问题 | 说明 | 建议 |
|------|------|------|
| **手机号 RSA 加密** | PROTOCOL 写“服务端提供的 RSA 公钥加密”，公钥如何下发、轮换未约定 | 实现时需约定：登录/enroll 响应中带公钥或单独接口获取、公钥过期与轮换策略 |
| **审计 Key 派生** | “Key 动态派生自设备硬件指纹”未约定算法 | **已约定**：PROTOCOL 第 6 节已明确 HKDF-SHA256、输入 device_id、info="mop.audit.v1"、输出 32 字节，salt 策略一致，各端须同算法 |
| **Wipe 原子性** | 自毁为多步操作，若中途进程被 Kill，可能处于半清空状态 | 建议约定：按顺序清空后再重启；或启动时检测“待完成 Wipe”并补全，避免残留敏感数据 |
| **用户须知与同意** | 规约要求首次登录展示《用户须知》并勾选“已阅读并同意”，内容与持久化方式未在规约中明确 | 建议约定：存储字段或本地 key、版本号，以便后续须知更新时再次征意 |

## 5. 多端与集成

| 问题 | 说明 | 建议 |
|------|------|------|
| **Web 与 api 鉴权对接** | web 登录后使用 api 的 auth/login，token、refresh、跨域需统一 | 提前约定 CORS、Token 放在 Header 名称、refresh 时机与接口，避免 web 请求被拒或 token 失效处理不一致 |
| **API 版本与兼容** | 多端（App / web / admin）并行开发时，/api/v1 变更可能影响旧客户端 | 建议约定版本策略：如 v1 仅追加字段不删改、或约定弃用周期，避免线上多版本行为混乱 |
| **同一 User 多 Device** | 远程指令按 DeviceID 下发，某设备长期离线时的限频、清理未约定 | 实现时可约定：离线超过 N 天是否仍可下发、是否在管理端标记“离线”并限制部分操作 |

## 7. 边界与异常行为

| 问题 | 说明 | 建议 |
|------|------|------|
| **扫码内容非法** | 非 mop 协议或 Base64 解析失败时的提示与是否落盘未写 | 建议约定：仅提示“无效二维码”，不写入 Host/凭证，避免污染本地配置 |
| **邀请链接已登录用户打开** | 已登录用户点击邀请链接时的行为未定义（跳转/忽略/提示） | 建议约定：跳转至“添加邀请关系”或提示“您已登录”，避免与未登录流程混淆 |
| **网络切换** | 从无网到有网、从有网到无网时的重连与“API 失效”判定时机 | 建议约定：重连间隔、何时再次判定失效、是否在恢复后自动重试 enroll 或登录 |

## 9. 应用运行过程可能遇到的问题（Runtime）

以下为**应用上线后、用户使用过程中**可能出现的运行期问题，实现时需约定行为或做容错，避免白屏、卡死、数据不一致或体验混乱。

| 类别 | 问题 | 说明 | 建议 |
|------|------|------|------|
| **网络与连接** | 长时间断网/弱网后恢复 | 连续 3 次失败后已提示“API 失效”并引导扫码；恢复网络后每 30s 自动重试（PROTOCOL 第 7 节）。若用户未扫码且一直等待，需避免重复弹窗或占用资源过度重试。 | 实现时约定：后台重试次数上限或静默重试 N 次后仅保留“手动重试”或“扫码激活”入口；弱网下超时 15s 可能过短，可按需做自适应或用户可感知的“正在重试”提示。 |
| **网络与连接** | Tinode 长连断线 | 长连断开后需重连；token 过期时需先 refresh 再连。若 refresh 失败则跳转登录。iOS 依赖推送唤醒，断线期间指令可能延迟。 | 实现时约定：重连间隔与退避、断线后是否拉取离线消息、推送未到达时下次前台时的补偿拉取；与 PROTOCOL 第 2.2、ARCHITECTURE 2.2 一致。 |
| **鉴权与会话** | token 过期且 refresh 失败 | 401 后尝试一次 refresh，失败则跳转登录（PROTOCOL 2.2）。用户可能丢失未提交的草稿或当前页状态。 | 实现时约定：跳转前是否提示“登录已过期，请重新登录”；敏感页（如编辑中）是否尝试本地暂存或提示保存。 |
| **鉴权与会话** | 同一 User 多端登录 | 多 Device 同时在线时，若服务端有踢人策略（如后登录踢先登录），未在规约中约定。 | 实现时约定：是否允许多端同时在线、是否踢人及踢人策略；若踢人，被踢端收到 401 或专用 code 后跳转登录并提示“已在其他设备登录”。 |
| **审计上报** | 上报失败与重试 | PROTOCOL 3.2：失败时指数退避重试（如 1min、2min、5min）。若服务端长期 5xx 或 4xx，客户端可能堆积重试任务。 | 实现时约定：最大重试次数或放弃条件、本地队列上限，避免占满存储或电量；服务端恢复后 check-sum 会再次拉取，可依赖下次冷启动/前台上报。 |
| **审计上报** | 服务端解密失败 | device_id 或 Key 派生不一致（如升级后 salt 策略变化）、包损坏会导致服务端解密失败。 | 实现时约定：服务端返回明确错误码（如 decrypt_failed），客户端不无限重试同一包；各端严格按 PROTOCOL 第 6 节 Key 派生，版本升级时 salt/算法兼容策略。 |
| **远程指令** | 设备离线或推送未到达 | 指令经 Tinode 下发（在线）或厂商推送（离线）。推送可能延迟、被系统限制或用户关闭通知权限，导致指令未执行。 | 实现时约定：管理端展示“已下发/待送达/超时”；离线指令可设置 TTL，超时后标记失败；用户恢复在线后 Tinode 重连，可考虑“拉取待执行指令”的补偿（若协议支持）。 |
| **远程指令** | 拨号/短信被用户取消或权限被关 | mop.cmd.dial / mop.cmd.sms 唤起系统拨号盘/短信界面，用户可取消；若用户事后撤销电话/短信权限，唤起失败。 | 实现时约定：唤起失败时是否上报失败状态给管理端、是否提示用户检查权限；非静默指令允许用户取消，规约已明确。 |
| **音视频** | Jitsi 房间不存在或已结束 | 信令收到后加入 Jitsi，若房间已解散或 jit 侧异常，加入失败。 | 实现时约定：加入失败时的提示（如“会议已结束”）、是否通知 Tinode 或发起方；避免无限重试加入。 |
| **音视频** | 网络切换或 jit 域不可用 | WiFi/4G 切换可能导致 Jitsi 断线；jit 域证书错误或不可用时客户端无法入会。 | 实现时约定：断线重连或提示“网络异常，请检查后重试”；证书错误时提示用户或记录日志，不静默吞掉。 |
| **本地与权限** | 用户事后撤销相册/通讯录/悬浮窗 | 规约要求未完全授予则锁定引导页；运行中用户去设置里撤销权限后，再次进入 App 或执行依赖该权限的功能时会失败。 | 实现时约定：进入主列表后若检测到权限被撤销，是否降级（如仅隐藏依赖该权限的入口）或再次引导至设置/引导页；保存凭证时相册不可用则提示与重试（.cursorrules 已要求）。 |
| **本地与权限** | SecureStorage 满或损坏 | Wipe 需清空 SecureStorage；若存储满或损坏，清空可能不完整。 | 实现时约定：启动时“待完成 Wipe”标记存在则继续清空（ARCHITECTURE 5.2）；清空失败时记录并尽量删除本地 DB/缓存，避免残留敏感数据。 |
| **服务端与依赖** | api / Tinode / jit 单点不可用 | 四域中某一域宕机或维护，用户侧表现为 API 失效、无法收发消息或无法入会。 | 实现时约定：按 PROTOCOL 第 7 节判定 API 失效并引导扫码/重试；Tinode 不可用时提示“服务暂时不可用”；运维侧保证高可用或维护窗口公告。 |
| **版本与兼容** | 旧版 App 连新版 api | 新版 api 新增必填字段或变更语义时，旧版 App 可能 4xx 或行为异常。 | 实现时约定：ARCHITECTURE 3.2 建议 v1 仅追加不删改；必填变更需版本协商或弃用周期，避免线上旧版大面积报错。 |
| **版本与兼容** | 须知版本号更新触发再次征意 | 服务端下发须知版本号高于本地已同意版本时，须再次展示须知并征意。若未同意前限制部分功能，需约定限制范围。 | 实现时约定：再次征意前是否允许继续使用（仅弹窗提示）还是限制敏感操作；与 .cursorrules 用户须知与再次征意一致。 |

## 方案矛盾排查结论

对规约、架构、协议、Tinode/Jitsi 集成、风险文档做了交叉比对，结论如下。

**已发现并修正的表述不一致**
- **DEVELOPMENT-RISKS 已收敛说明**：原写「Token 刷新（可选）」与 PROTOCOL 第 2.2 节「常规方案、提供 refresh 接口」不一致，已改为「常规方案：提供 refresh 接口，401 时先尝试一次 refresh…」。
- **.cursorrules 语言规范**：原「所有 UI 文案、代码注释、文档必须使用简体中文」易与「i18n 支持英文」读成冲突。已改为：**代码注释与文档须使用简体中文；UI 文案以简体中文为源语言，多语言由 i18n 提供翻译**，与开发期仅中英两语种一致。

**未发现实质性逻辑矛盾**
- 激活顺序、enroll 分配 UID、邀请与互为好友、管理数据来自设备/用户经 api 汇总、device_id 常规方案、音视频信令以 Tinode 为准、审计 Key 与 device_id、Wipe 与启动补全、凭证 timestamp、错误码与重试等，各文档表述一致。
- 登录响应中「可选 refresh_token」与「常规方案提供 refresh 接口」兼容：即登录可返回或不返回 refresh_token，若返回则 api 提供 refresh，客户端 401 时尝试一次 refresh。

**建议**
- 后续修订时，凡涉及「可选/必选」「须/建议」的变更，同步检查 PROTOCOL、.cursorrules、ARCHITECTURE 三处，避免再次出现「已收敛说明」与协议正文不一致。
