# MOP 项目可行性检查

本文档从技术、依赖、平台、合规与上架、资源与节奏等维度对项目可行性进行集中检查，结论与前提条件供立项与评审参考。详细风险与已收敛说明见 **DEVELOPMENT-RISKS.md**。

---

## 1. 技术可行性

| 维度 | 结论 | 说明 |
|------|------|------|
| **前端与跨端** | 可行 | Flutter 支持 Android 10+ / iOS 17+，UI、i18n、SecureStorage、Isolate 等能力满足规约；用户端网页可为 Flutter Web 或独立 SPA，与 api 账密登录对接无技术障碍。 |
| **IM 与长连接** | 可行 | Tinode 为开源 IM 服务，支持 WebSocket、消息与订阅模型；与 REST api 并存、Host 统一，技术路径清晰。 |
| **音视频与屏幕共享** | 可行 | Jitsi 开源、支持自建与 HTTPS；信令经 Tinode、媒体走 Jitsi，TINODE-JITSI-INTEGRATION 已约定房间命名、昵称入会与客户端接入方式，Flutter/Web 均可通过 SDK 或 WebView 嵌入。 |
| **原生能力** | 可行 | 通讯录、短信、通话记录、UsageStats、相册、拨号/短信唤起、Android 悬浮窗在对应平台有稳定 API；iOS 已约定仅使用平台允许的采集项，避免不可实现项。 |
| **后端与四域** | 可行 | api / admin / web / jit 四域、Docker Compose 编排、REST + WebSocket + 管理端接口（PROTOCOL 5.1）均为成熟方案；dev-env 已提供 hosts、证书、前置脚本与 Compose 占位。 |
| **安全与加密** | 可行 | TLS 1.2/1.3、RSA 公钥下发、AES-256-GCM 审计上报、Key 由设备指纹派生、Wipe 与启动补全等均在规约与 PROTOCOL 中约定，无未解技术难点。 |

**小结**：技术栈与端到端流程均可落地，无单点不可实现项。

## 2. 依赖与生态可行性

| 维度 | 结论 | 说明 |
|------|------|------|
| **Tinode** | 可行 | 开源、可自部署，与 api 同域或内网互通；消息与扩展满足信令与指令通道需求。 |
| **Jitsi** | 可行 | 开源、可自建 jit 域；Meet API / SDK 支持 Web 与移动端嵌入，屏幕共享与昵称入会可实现。 |
| **Flutter** | 可行 | 官方支持 Android/iOS，插件生态含 secure_storage、QR、WebView 等；原生桥接（Method Channel）为官方推荐方式。 |
| **推送** | 可行 | FCM（Android）、APNs（iOS）、厂商推送均有成熟集成方式，用于静默唤醒与指令补偿链路。 |

**小结**：核心依赖均为可获取、可自控的开源或标准服务，无强依赖不可控第三方闭源组件。

## 3. 平台可行性（Android / iOS）

| 维度 | 结论 | 说明 |
|------|------|------|
| **最低版本** | 可行 | Android 10 (API 29) / iOS 17.0 为规约底线，市场占比与官方支持均充足。 |
| **审计采集** | 可行 | Android 可覆盖通讯录、短信、通话、App 列表、UsageStats；iOS 仅实现平台允许项（如通讯录），短信/通话/UsageStats 等已明确不实现、不展示、不上报，规约与 PROTOCOL 已区分。 |
| **保活与后台** | 可行 | Android Foreground Service + 业务伪装通知；iOS 依赖系统后台与 APNs 唤醒，ARCHITECTURE 已约定，无超出系统能力的要求。 |
| **权限与相册** | 可行 | 相册、通讯录、悬浮窗（仅 Android）均为标准权限与 API；未授予则锁定引导页，逻辑清晰。 |

**小结**：双端能力边界已约定，iOS 限制已显式写入规约，不构成可行性障碍。

## 4. 合规与上架可行性

| 维度 | 结论 | 说明 |
|------|------|------|
| **隐私与告知** | 已约定 | 用户须知首次展示、勾选同意与登录/注册一并提交、版本号与同意状态持久化、再次征意触发条件均已写入 .cursorrules；手机号对用户不可见、后端脱敏、管理端可见范围明确。 |
| **数据最小化与安全** | 已约定 | 审计数据加密上报、Key 派生一致、Wipe 与启动补全、无明文落地等见 PROTOCOL 与 ARCHITECTURE；实现时需落实访问控制与存储策略。 |
| **应用商店与政策** | 需前置确认 | 影子审计、远程指令（含拨号/短信/自毁）等属于敏感能力，**上架前建议**：① 确认目标应用商店（如 Google Play、App Store、企业分发）对后台数据采集与远程管控的合规要求；② 必要时进行隐私与合规评审，确保《用户须知》与权限说明完整、准确。规约已要求须知与权限引导，实现与上架时需与法务/产品对齐。 |

**小结**：产品与规约层面已做隐私与告知约定；**应用商店与属地法律合规**需在正式上架前单独确认，属实施前提而非技术不可行。

## 6. 总体结论与前提条件

**总体结论**：在当前规约与架构下，项目**具备可行性**，技术、依赖、平台、合规约定与资源节奏均可支持实施。

**前提条件（建议在实施/上架前确认）**：

1. **应用商店与合规**：根据目标分发渠道（如 Google Play、App Store、企业内部分发）确认对后台数据采集、远程指令、隐私告知与权限说明的要求，必要时完成法务/合规评审。
2. **须知版本号**：再次征意依赖“当前须知版本”；实现时约定版本号由服务端下发（如 config）还是客户端写死，并各端一致（见 DEVELOPMENT-RISKS 检查清单）。
3. **实现期约定**：超时与重试阈值、错误码表、审计包格式细节等可在实现阶段在 PROTOCOL 或 api 文档中固定，按 ARCHITECTURE 第 11 节与开发阶段逐项收敛。
